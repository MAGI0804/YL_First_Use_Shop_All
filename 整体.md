# 悠兰儿童商城项目整体说明与后续开发规划

生成日期：2026-05-29  
项目路径：`D:\youlan_kids_shop_self`  
目标：梳理当前小程序、Web 管理端、后端整体内容，并给出后续上线闭环方向、开发任务与人天估算。

## 1. 项目整体概览

本项目由三部分组成：

| 模块       | 目录                  | 技术栈                                             | 主要职责                                                     |
| ---------- | --------------------- | -------------------------------------------------- | ------------------------------------------------------------ |
| 小程序     | `wx_ui_kids`          | 微信原生小程序                                     | 面向用户的商品浏览、搜索、购物车、下单、订单、地址、售后、消息、活动展示 |
| Web 管理端 | `Management_web_shop` | Vue 3 + Vite + TypeScript + Element Plus + ECharts | 面向运营/管理人员的商品、订单、会员、售后、活动、报表管理    |
| 后端       | `Member_Shop`         | Go + Gin + GORM + MySQL + Redis                    | 提供商品、库存、订单、购物车、地址、售后、评价、数据分析、活动、消息等 API |

当前项目已有较完整的业务雏形，后端接口覆盖面较广，小程序页面链路较全，Web 管理端已经有主要页面框架。但距离稳定上线仍需优先补齐构建、配置、安全、测试、交易闭环和运维能力。

### 当前问题

| 问题                             | 影响                                       |
| -------------------------------- | ------------------------------------------ |
| 生产域名硬编码                   | 开发、测试、生产环境切换困难               |

### 验证结果

后端执行：

`go test ./...`

结果：通过。

但测试覆盖较少，只有少量测试包，核心业务如订单、库存、售后、商品、token 中间件缺少系统性测试。

### 当前问题

| 问题                                             | 影响                                             |
| ------------------------------------------------ | ------------------------------------------------ |
| 数据库、Redis、JWT、微信、短信等存在默认敏感配置 | 有安全风险，不能直接作为生产交付状态             |
| `AutoMigrate` 启动时自动迁移                     | 生产环境需要更可控的迁移策略                     |
| access_token 与 IP/Redis 绑定逻辑复杂            | 可能造成客户端 token 失效、代理 IP、并发访问问题 |
| CORS 当前放开 `*`                                | 生产环境需收敛                                   |
| 接口返回格式不完全统一                           | 前端需要兼容多种错误结构                         |
| 聚水潭库存同步、流量分析仍为预留                 | 上线范围需要明确                                 |
| 日志路径写死倾向明显                             | 部署环境需统一日志目录和权限                     |

## 5. 核心风险

| 优先级 | 风险                     | 建议                                  |
| ------ | ------------------------ | ------------------------------------- |
| P0     | Web 构建失败             | 先修复工具链和 `tsconfig`，确保可打包 |
| P0     | 敏感配置硬编码           | 全部改为环境变量，并更新部署说明      |
| P0     | 生产/测试环境混用        | 小程序、Web、后端统一环境配置         |
| P0     | 支付链路未明确           | 明确接入微信支付或先做模拟支付闭环    |
| P1     | 订单、库存、售后缺少测试 | 补核心业务测试，避免上线后数据错乱    |
| P1     | 中文编码问题             | 统一 UTF-8，修复乱码文件              |
| P1     | 接口返回结构不统一       | 制定统一响应结构和错误码              |
| P2     | 运维说明不足             | 补部署、日志、数据库迁移、回滚方案    |
| P2     | 后台权限较弱             | 后续补角色权限和操作审计              |

## 6. 后续开发方向

后续建议按“补齐上线闭环”推进，而不是先做新功能扩展。

### 第一阶段：上线基础修复

目标：项目可以稳定构建、配置清晰、安全风险可控。

重点：

- 修复 Web 构建
- 补齐 TypeScript 配置
- 抽离环境变量
- 清理敏感默认值
- 统一 API 域名配置
- 整理启动、部署和接口文档

### 第二阶段：交易主链路闭环

目标：用户从浏览商品到下单、支付、订单流转、售后可完整跑通。

重点：

- 商品浏览和上下架状态
- 库存展示与下单校验
- 购物车和订单确认
- 支付接入或支付占位
- 订单状态流转
- 售后申请、审核、收货、库存回滚

### 第三阶段：运营后台可用

目标：运营人员能通过后台完成日常管理。

重点：

- 商品管理
- 活动图管理
- 订单发货
- 售后审核
- 会员查询
- 数据报表
- 图片上传和资源管理

### 第四阶段：质量和运维增强

目标：降低维护成本和线上事故风险。

重点：

- 后端核心测试
- 前端冒烟测试
- 统一错误码
- 日志和监控
- 数据库迁移策略
- 备份和回滚流程

## 8. 推荐排期

| 周期    | 工作内容                              |
| ------- | ------------------------------------- |
| 第 1 周 | 修复构建、配置、安全、文档、环境切换  |
| 第 2 周 | 商品、购物车、下单、订单状态主链路    |
| 第 3 周 | 支付/支付占位、售后闭环、后台订单售后 |
| 第 4 周 | 测试、部署、回归、上线检查、问题修复  |

如果 2 名开发并行，预计 3 到 4 周完成第一版上线闭环；如果 1 名开发独立推进，预计 5 到 6 周更稳妥。

## 9. 建议优先级

### 必须先做

1. 修复 Web 构建失败。
2. 清理敏感配置和环境变量。
3. 明确支付方案。
4. 跑通商品、购物车、下单、订单、售后主链路。
5. 补后端核心测试。

### 可以后做

1. 报表精细化。
2. 流量分析。
3. 更复杂的会员运营能力。
4. 后台角色权限细分。
5. 自动化发布流水线。

## 10. 结论

当前项目不是从零开始，三端核心业务框架已经具备，后端接口覆盖较完整，小程序页面链路较丰富，Web 后台也有主要管理页面。

但目前还不适合直接进入生产上线，主要阻塞是 Web 构建失败、敏感配置硬编码、环境配置不统一、支付链路未明确、核心测试不足、编码和文案问题。建议后续不要先扩新功能，而是优先补齐上线闭环，把交易主流程和后台运营流程稳定下来。
