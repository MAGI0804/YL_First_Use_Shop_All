# 会员与订单体系完善开发文档

生成日期：2026-06-05  
适用范围：`Member_Shop` 后端、`Management_web_shop` 管理端、`wx_ui_kids` 小程序端  
目标模块：会员体系、后台代下单、购物车管理、订单查询、会员详情、会员标签

## 1. 背景与目标

当前项目已经具备商品、购物车、订单、会员基础模型和部分后台页面，但会员管理仍不完整：

- 后端已有 `member_info` 表，包含会员号、手机号、订单总金额、已支付金额、天猫/有赞字段。
- 管理端已有 `Member.vue`、`MemberDetail.vue` 路由和页面，但页面仍以本地模拟数据为主，未完整接入真实会员接口。
- 购物车已有 `/cart/add_to_cart`、`/cart/query_cart_items`、`/cart/clear_cart` 等接口，但后台缺少面向运营的会员购物车查看、增删商品和代下单流程。
- 订单列表已有订单号模糊查询 `tid`，但未完整支持手机号、会员 ID、子订单号等运营查询入口。
- 会员标签在管理端只是前端假数据，后端没有独立标签模型和查询能力。

本次开发目标是补齐后台运营需要的会员与订单闭环：

1. 后台能帮会员下单。
2. 会员新增一个运营手动输入的唯一字段。
3. 后台能查询会员购物车，后台下单后按下单商品清除购物车余量。
4. 后台能直接为用户新增购物车商品。
5. 会员能展示和统计总下单金额、已支付金额。
6. 会员新增天猫 ID、天猫金额、有赞 ID、有赞金额。
7. 订单查询支持手机号、会员 ID、子订单号。
8. 新增并接入真实会员详情页。
9. 支持自定义会员标签，会员查询可按标签筛选。

## 3. 业务口径

### 3.1 会员身份

会员身份以 `member_info` 为准，微信用户 `users_user` 只表示小程序登录身份。

- `member_info.user_id`：关联小程序用户 ID，可为空或为 0，适用于后台先建会员、后续再绑定微信的场景。
- `member_info.mobile`：会员手机号，继续保持唯一。
- `member_info.member_no`：系统生成或兼容旧会员号，继续保持唯一。
- 新增 `manual_unique_code`：运营手动输入的唯一字段，必须唯一，允许作为线下会员编号、CRM 编号或门店客户编号使用。

建议命名：`manual_unique_code`。  
不建议直接复用 `member_no`，因为当前 `member_no` 有自动生成逻辑，手动字段应和系统字段分开，避免导入或绑定时冲突。

### 3.2 订单金额

会员金额统计使用两个口径：

- `total_order_amount`：会员总下单金额，创建订单成功后按 `order_amount` 累加。
- `total_paid_amount`：会员已支付金额，运营确认支付后按 `final_pay_amount` 累加。

注意：

- 后台代下单创建成功，也应增加 `total_order_amount`。
- 订单取消后是否扣回 `total_order_amount` 需要产品确认。建议本期保持“总下单金额表示历史成功提交订单金额”，取消不扣回；报表如需净成交额使用订单状态过滤。
- 已支付金额只在 `pay_status` 从 `unpaid` 变为 `paid` 时累加，避免重复确认支付导致重复累计。

### 3.3 购物车余量

后台代下单可选择来自购物车的商品，也可直接录入商品。

当代下单商品来自购物车时，下单成功后按 SKU 扣减购物车数量：

- 购物车数量 > 下单数量：保留商品，数量减去下单数量。
- 购物车数量 = 下单数量：删除该 SKU。
- 购物车数量 < 下单数量：下单前拦截，提示购物车余量不足。
- 商品不在购物车但请求标记为 `from_cart=true`：下单前拦截。

如果代下单请求为直接商品录入，不清购物车。

```go
    Module         string    `gorm:"column:module;size:50;index;not null" json:"module"`
    TargetType     string    `gorm:"column:target_type;size:50;index;not null" json:"target_type"`
    TargetID       string    `gorm:"column:target_id;size:100;index;not null" json:"target_id"`
    MemberID       uint      `gorm:"column:member_id;index" json:"member_id"`
    UserID         int       `gorm:"column:user_id;index" json:"user_id"`

    AfterData      string    `gorm:"column:after_data;type:json" json:"after_data"`

    ClientIP       string    `gorm:"column:client_ip;size:64" json:"client_ip"`
    UserAgent      string    `gorm:"column:user_agent;size:255" json:"user_agent"`
    Remark         string    `gorm:"column:remark;type:text" json:"remark"`
    CreatedAt      time.Time `gorm:"column:created_at;autoCreateTime" json:"created_at"`
}
```

必须记录的动作：

| 动作编码 | 模块 | 说明 |
| --- | --- | --- |
| `member.create` | `member` | 后台新增会员 |
| `member.update` | `member` | 后台编辑会员基础信息、手动唯一字段、外部平台字段 |
| `member.status.update` | `member` | 启用、停用会员 |
| `member.tag.set` | `tag` | 覆盖会员标签 |
| `member.tag.add` | `tag` | 追加会员标签 |
| `member.cart.add` | `cart` | 后台为会员加入购物车商品 |
| `member.cart.update_quantity` | `cart` | 后台修改购物车商品数量 |
| `member.cart.delete` | `cart` | 后台删除购物车商品 |
| `member.cart.clear_after_order` | `cart` | 后台代下单成功后扣减购物车余量 |
| `order.backend_create` | `order` | 后台代会员下单 |
| `order.payment_amount.update` | `order` | 后台修改订单实付金额 |
| `order.payment.confirm` | `order` | 后台确认支付 |
| `order.cancel` | `order` | 后台取消订单 |

记录规则：

- 操作人必须从后台登录 token 或后端 session 解析，不能只信任前端传入 `operator_id`。
- 同一个后台请求如果影响多个对象，可以写多条日志；例如后台代下单成功后至少写 `order.backend_create` 和 `member.cart.clear_after_order`。
- `before_data` 和 `after_data` 只记录关键业务字段，不记录密码、token、验证码等敏感信息。
- 操作失败时，如果已经进入业务事务但最终回滚，可记录一条失败日志到独立事务，`action` 后缀可使用 `.failed`，例如 `order.backend_create.failed`。
- 日志写入失败不能静默忽略。核心操作建议事务内写日志；如果日志写入失败，应返回错误或至少写服务日志并报警。

## 5. 后端接口设计

响应结构沿用当前项目 `msg.SuccessResponse` / `msg.ErrResponseStr` 风格。

### 5.1 会员管理接口

建议路由前缀：`/member/`。  
也可以继续放在 `/ordinary_user/` 下，但后台会员管理和小程序用户接口职责不同，建议新建 `member_route.go`、`member_controller.go`。

#### 5.1.1 新增会员

`POST /member/create`

请求：

```json
{
  "mobile": "13800000000",
  "nickname": "张三",
  "manual_unique_code": "CRM-0001",
  "tmall_id": "tm001",
  "tmall_amount": 100.00,
  "youzan_id": "yz001",
  "youzan_amount": 50.00,
  "source": "backend",
  "remarks": "线下客户"
}
```

规则：

- `mobile` 必填且唯一。
- `manual_unique_code` 非空时必须唯一。
- 如果手机号已存在会员，返回冲突提示，不覆盖原数据。
- 新增会员不强制关联 `users_user`；如果同手机号已有 `users_user`，可自动绑定 `user_id`。

#### 5.1.2 更新会员

`POST /member/update`

请求：

```json
{
  "id": 1,
  "nickname": "张三",
  "manual_unique_code": "CRM-0001",
  "tmall_id": "tm001",
  "tmall_amount": 100.00,
  "youzan_id": "yz001",
  "youzan_amount": 50.00,
  "status": "active",
```

- 手机号原则上不允许直接修改；如必须支持，需要单独接口并记录操作日志。

- 金额字段不允许为负数。

#### 5.1.4 会员详情

`POST /member/detail`

请求：

```json
{
  "id": 1,
  "user_id": 10001,
  "member_no": "M202606050001",
  "mobile": "13800000000"
}
```

规则：

- 至少传一个查询条件。
- 优先级：`id` > `user_id` > `member_no` > `mobile`。

返回包含：

- 基础信息：会员号、手动唯一字段、手机号、昵称、状态、来源、备注。
- 外部平台：天猫 ID/金额、有赞 ID/金额。
- 金额统计：总下单金额、已支付金额、待支付金额。
- 标签列表。
- 最近订单，可限制 10 条。
- 当前购物车摘要。

#### 5.1.5 会员金额汇总

现有 `/ordinary_user/member_amount_summary` 可保留。  
建议新增后台语义接口：

`POST /member/amount_summary`

请求同会员详情查询条件，返回：

```json
{
  "total_order_amount": 300.00,
  "total_paid_amount": 200.00,
  "pending_pay_amount": 100.00,
  "tmall_amount": 100.00,
  "youzan_amount": 50.00
}
```

`pending_pay_amount` 可实时计算：

```sql
SUM(final_pay_amount) WHERE user_id = ? AND pay_status <> 'paid' AND status <> 'canceled'
```

### 5.4 后台代下单接口

#### 5.4.1 创建后台订单

`POST /order/backend_create_order`

请求：

```json
{
  "member_id": 1,
  "user_id": 10001,
  "mobile": "13800000000",
  "receiver_name": "张三",
  "receiver_phone": "13800000000",
  "province": "浙江省",
  "city": "杭州市",
  "county": "西湖区",
  "detailed_address": "文一路1号",
  "product_list": [
    {
      "commodity_id": "SKU001",
      "quantity": 2,
      "from_cart": true
    },
    {
      "commodity_id": "SKU002",
      "quantity": 1,
      "from_cart": false
    }
  ],
  "order_amount": 299.00,
  "remark": "后台代下单",
  "operator_id": 1
}
```

处理流程：

1. 解析会员，优先使用 `member_id`，其次 `user_id`、`mobile`。
2. 校验会员存在、状态可用。
3. 解析或创建 `user_id`。
4. 校验收货人、手机号、地址。
5. 校验商品存在、数量大于 0。
6. 如果 `from_cart=true`，锁定购物车并校验购物车余量。
7. 使用现有 `CreateOrder` 逻辑创建订单、子订单、扣库存。
8. 写入后台代下单字段：`created_by_backend=true`、`backend_operator_id`、`backend_order_remark`。
9. 下单成功后，在同一事务内扣减购物车余量。
10. 累加会员 `total_order_amount`。
11. 写入操作日志：`order.backend_create`。
12. 如果扣减了购物车，写入操作日志：`member.cart.clear_after_order`，记录扣减前后的购物车数量。
13. 返回订单号、子订单号、剩余购物车。

关键事务要求：

- 创建订单、扣库存、扣购物车余量、会员金额累加必须在同一事务内完成。
- 任何一步失败均回滚。
- 购物车 JSON 更新需要对 `cart_cart` 行加行锁，避免运营和用户同时改购物车造成余量错误。
- 操作日志必须记录真实后台操作人，后台操作人从 token 解析；请求体中的 `operator_id` 只作为兼容字段。

返回：

```json
{
  "order_id": "Y2026060500000001",
  "sub_order_ids": ["S2026060500000001"],
  "status": "pending",
  "pay_status": "unpaid",
  "order_amount": 299.00,
  "final_pay_amount": 299.00,
  "cart_remaining": [
    {
      "commodity_code": "SKU001",
      "quantity": 1
    }
  ]
}
```

### 5.5 订单查询扩展

扩展现有 `POST /order/orders_query`。

请求新增字段：

```json
{
  "shopname": "youlan_kids",
  "page": 1,
  "page_size": 20,
  "tid": "Y20260605",
  "mobile": "13800000000",
  "member_id": 1,
  "member_no": "M202606050001",
  "sub_order_id": "S2026060500000001",
  "status": "pending",
  "pay_status": "unpaid",
  "begin_time": "2026-06-01",
  "end_time": "2026-06-05"
}
```

查询规则：

- `tid`：继续按 `order_id LIKE` 查询。
- `mobile`：同时匹配 `order_data.receiver_phone` 和 `member_info.mobile`。建议通过 `member_info.user_id` 补充查询订单。
- `member_id`：先查 `member_info.id` 对应 `user_id`，再查 `order_data.user_id`。
- `member_no`：先查 `member_info.member_no` 对应 `user_id`。
- `sub_order_id`：先查 `sub_order_data.sub_order_id` 对应 `order_id`，再过滤主订单。
- `pay_status`：新增支付状态过滤。

建议 SQL 逻辑：

```sql
SELECT DISTINCT o.*
FROM order_data o
LEFT JOIN member_info m ON m.user_id = o.user_id
LEFT JOIN sub_order_data so ON so.order_id = o.order_id
WHERE
  (:tid = '' OR o.order_id LIKE CONCAT('%', :tid, '%'))
  AND (:mobile = '' OR o.receiver_phone = :mobile OR m.mobile = :mobile)
  AND (:member_id = 0 OR m.id = :member_id)
  AND (:member_no = '' OR m.member_no = :member_no)
  AND (:sub_order_id = '' OR so.sub_order_id = :sub_order_id)
```

返回字段建议增加：

- `member_id`
- `member_no`
- `member_mobile`
- `manual_unique_code`
- `sub_orders` 或至少 `sub_order_ids`

## 8. 权限与安全

后台新增能力必须要求后台登录态：

- 查询会员列表、详情：运营、客服、管理员可用。
- 新增/编辑会员：运营、管理员可用。
- 编辑会员标签：运营、管理员可用；客服可只读或按权限配置。
- 后台加入购物车、代下单：运营、管理员可用。
- 修改外部平台金额：管理员或运营主管权限，避免普通客服误改金额。

后端实现建议：

- 新增接口统一通过后台 token 中间件校验。
- `operator_id` 优先从 token 解析，不信任前端传入值；前端传入只作为兼容。
- 后台所有写操作必须写入 `backend_operation_log`，不能只写业务表。
- 会员新增、会员编辑、标签修改、后台加购物车、后台代下单、购物车余量扣减、订单改价、确认支付、取消订单都必须记录操作人和操作前后关键数据。
- 日志中的操作人必须来自后台 token 解析结果，前端传入的 `operator_id` 不能作为唯一依据。
- 管理端页面不允许手填“操作人 ID”。当前订单改价、确认支付弹窗中已有操作人 ID 输入框，后续应改为只读展示当前登录人，提交时由后端按 token 绑定操作人。

后台还需要提供操作记录查询能力，便于会员详情和订单详情展示审计轨迹。

## 11. 风险与注意事项

1. 购物车当前是 JSON 字段，局部扣减需要整段读写，必须加行锁，否则并发下可能覆盖用户新加购数据。
2. 会员没有绑定 `user_id` 时，后台加购物车和订单展示会受影响。建议后台新增会员时可创建占位用户，或在代下单时自动创建。
3. 会员总下单金额和已支付金额要防止重复累加，尤其是重复确认支付、订单状态回退、手动修复数据。
4. 订单查询新增 `LEFT JOIN` 后要注意分页去重，建议用 `DISTINCT order_id` 或先查订单号再查详情。
5. 标签名称需要统一去空格和长度限制，避免出现看似相同的标签。
6. 管理端现有会员页是假数据，接接口时需要清理本地筛选逻辑，改为服务端分页查询。

## 14. 推荐开发顺序

1. 先做会员模型、会员列表/详情、标签接口。
2. 再把管理端会员列表和详情页从模拟数据切到真实接口。
3. 接后台购物车查询和加购。
4. 做后台代下单和购物车余量扣减事务。
5. 接入操作日志查询，在会员详情和订单详情展示操作记录。
6. 最后扩展订单查询条件和订单页筛选。

这样能先让会员数据真实可见，再接交易操作，避免后台代下单开发时缺少会员详情入口和排查页面。
